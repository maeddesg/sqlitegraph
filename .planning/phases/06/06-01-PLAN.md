# Key Context

**Phase 5 Achievements:**
- `HnswIndex::with_persistent_storage()` creates indexes with SQLite vector storage
- `SqliteGraph::load_hnsw_indexes()` auto-loads indexes on construction
- `SQLiteVectorStorage` persists vectors to `hnsw_vectors` table
- Database schema v3: `hnsw_indexes`, `hnsw_vectors` tables exist

**Current Problem (from hnsw_cli_persistence_issue_20241223.md):**
- CLI's `hnsw-create` uses `graph.hnsw_index()` which calls `HnswIndex::new()` (in-memory)
- Vectors don't persist to database because index uses `InMemoryVectorStorage`
- Each CLI invocation creates new `SqliteGraph`, loads existing indexes, but new indexes are in-memory only

**Solution Approach:**
- Modify `SqliteGraph::hnsw_index()` to use `with_persistent_storage()` for file-based databases
- Keep in-memory storage for `:memory:` databases
- Add `is_persistent` helper to detect database type

**Key Files:**
- `sqlitegraph/src/hnsw/index.rs:1048-1074` - `hnsw_index()` method (uses `HnswIndex::new()`)
- `sqlitegraph/src/hnsw/index.rs:147-165` - `with_persistent_storage()` method (needs `Connection`)

Add `hnsw_index_persistent()` method to SqliteGraph that creates indexes with persistent storage for file-based databases:

```rust
impl SqliteGraph {
    /// Create or get an HNSW index with persistent storage (for file-based databases)
    ///
    /// This method automatically detects if the database is file-based and creates
    /// the index with SQLiteVectorStorage for automatic vector persistence.
    /// For in-memory databases, falls back to in-memory storage.
    ///
    /// # Arguments
    /// * `name` - Name to identify this index
    /// * `config` - HNSW configuration parameters
    pub fn hnsw_index_persistent(
        &self,
        name: &str,
        config: HnswConfig,
    ) -> Result<std::sync::RwLockWriteGuard<'_, HashMap<String, HnswIndex>>, SqliteGraphError>
    {
        use std::sync::RwLock;
        use rusqlite::Connection;

        // Check if index already exists
        {
            let indexes = self.hnsw_indexes.read().map_err(|e| SqliteGraphError::invalid_input(format!("RwLock poisoned: {}", e)))?;
            if indexes.contains_key(name) {
                return Err(SqliteGraphError::invalid_input(format!("HNSW index '{}' already exists", name)));
            }
        }

        // Check if database is file-based (not in-memory)
        let is_file_based = !is_in_memory_connection(&self.conn);

        // Create index with appropriate storage backend
        let hnsw = if is_file_based {
            // For file-based databases, use persistent storage
            // Clone connection to pass to with_persistent_storage
            let conn_for_storage = self.conn.clone()
                .map_err(|e| SqliteGraphError::invalid_input(format!("Failed to clone connection: {}", e)))?;
            HnswIndex::with_persistent_storage(name, config, conn_for_storage)
                .map_err(|e| SqliteGraphError::invalid_input(format!("Failed to create HNSW index: {}", e)))?
        } else {
            // For in-memory databases, use in-memory storage
            HnswIndex::new(name, config)
                .map_err(|e| SqliteGraphError::invalid_input(format!("Failed to create HNSW index: {}", e)))?
        };

        // Store the index (metadata already saved by with_persistent_storage)
        let mut indexes = self.hnsw_indexes.write().map_err(|e| SqliteGraphError::invalid_input(format!("RwLock poisoned: {}", e)))?;
        indexes.insert(name.to_string(), hnsw);

        Ok(indexes)
    }
}
```

**Key points:**
- Reuse existing `is_in_memory_connection()` helper (make pub or use internally)
- Use `Connection::clone()` to get a connection for `SQLiteVectorStorage`
- `with_persistent_storage()` already calls `save_metadata()`, so no need to call again
- Falls back to in-memory storage for `:memory:` databases
- Returns `RwLockWriteGuard<HashMap<String, HnswIndex>>` for consistency with existing API

**Why new method instead of modifying `hnsw_index()`:**
- Existing `hnsw_index()` may be used programmatically where in-memory is desired
- New method makes behavior explicit via naming
- Backward compatible - doesn't break existing code

```rust
#[test]
fn test_cli_hnsw_persistence_workflow() {
    use std::fs;
    use std::process::Command;
    use serde_json::Value;

    let db_path = "/tmp/test_cli_persist.db";
    let vectors_path = "/tmp/test_vectors.json";

    // Cleanup
    let _ = fs::remove_file(db_path);
    let _ = fs::remove_file(vectors_path);

    // Create test vectors file
    let vectors_json = r#"[
        {"vector": [1.0, 0.0, 0.0], "metadata": {"label": "a"}},
        {"vector": [0.0, 1.0, 0.0], "metadata": {"label": "b"}},
        {"vector": [0.0, 0.0, 1.0], "metadata": {"label": "c"}}
    ]"#;
    fs::write(vectors_path, vectors_json).unwrap();

    // Step 1: Create index
    let output = Command::new("cargo")
        .args(["run", "-p", "sqlitegraph-cli", "--",
                "--db", db_path,
                "hnsw-create", "--dimension", "3", "--m", "16",
                "--ef-construction", "200", "--distance-metric", "cosine"])
        .output()
        .expect("Failed to execute hnsw-create");
    assert!(output.status.success());

    // Step 2: Insert vectors (separate CLI invocation)
    let output = Command::new("cargo")
        .args(["run", "-p", "sqlitegraph-cli", "--",
                "--db", db_path,
                "hnsw-insert", "--input", vectors_path])
        .output()
        .expect("Failed to execute hnsw-insert");
    assert!(output.status.success());

    // Step 3: Verify stats (another CLI invocation)
    let output = Command::new("cargo")
        .args(["run", "-p", "sqlitegraph-cli", "--",
                "--db", db_path,
                "hnsw-stats"])
        .output()
        .expect("Failed to execute hnsw-stats");
    assert!(output.status.success());

    let stats: Value = serde_json::from_slice(&output.stdout).unwrap();
    assert_eq!(stats["vector_count"], 3);

    // Cleanup
    let _ = fs::remove_file(db_path);
    let _ = fs::remove_file(vectors_path);
}
```

This test validates:
- Index creation persists
- Vector insertion persists
- Index loads on subsequent CLI invocations
- Stats show correct vector count

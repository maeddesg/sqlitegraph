# Research Summary (from 54-RESEARCH.md)
- DPO (Double Pushout) is standard for graph rewriting: pattern -> interface -> replacement
- Simplified DPO: delete pattern nodes (except interface), add replacement nodes, rewire edges
- Interface nodes preserve connectivity between pattern and replacement
- Gluing condition ensures rewrite is well-formed (no dangling edges)
- For program analysis, simplified delete+insert is often sufficient (defer full DPO to pushout crate if needed)

# Key Technical Decisions
- Use simplified DPO: find matches with find_subgraph_patterns(), then apply deletion+insertion
- RewriteRule contains: pattern (SqliteGraph), replacement (SqliteGraph), interface (Vec<NodeId>)
- Interface nodes are preserved between pattern and replacement (indices into pattern/replacement)
- RewriteBounds: max_matches (default 10) to prevent explosion
- RewriteResult: rewritten_graph (SqliteGraph), patterns_replaced (usize), operations_applied (Vec<String>)
- Validation: after rewrite, verify no dangling edges (all edges reference valid nodes)

# Anti-Patterns to Avoid
- Rewriting without interface checks can create dangling edges - always validate
- Don't use pushout crate initially - simplified approach is sufficient for ML-02
- Always bound number of rewrites to prevent runaway transformations

    2. Main function rewrite_graph_patterns():
       Signature:
       ```rust
       pub fn rewrite_graph_patterns(
           graph: &SqliteGraph,
           rule: &RewriteRule<&str, &str>,
           bounds: RewriteBounds,
       ) -> Result<RewriteResult, SqliteGraphError>
       ```

       Algorithm:
       a) Convert rule.pattern to SqliteGraph for pattern matching
       b) Use find_subgraph_patterns() to find all pattern matches
       c) For each match (up to max_matches):
          - Create new graph by cloning original
          - Delete non-interface pattern nodes from matched locations
          - Add replacement nodes (except interface nodes which already exist)
          - Add replacement edges
          - Record operations in Vec<RewriteOperation>
       d) If validate_after_rewrite, check for dangling edges
       e) Return RewriteResult with rewritten graph and operations

    3. Progress variant rewrite_graph_patterns_with_progress():
       Signature:
       ```rust
       pub fn rewrite_graph_patterns_with_progress(
           graph: &SqliteGraph,
           rule: &RewriteRule<&str, &str>,
           bounds: RewriteBounds,
           progress: ProgressCallback,
       ) -> Result<RewriteResult, SqliteGraphError>
       ```

       Report progress:
       - "Found N pattern matches"
       - "Applying rewrite N/M"
       - "Validating rewritten graph"

    4. Validation function validate_no_dangling_edges():
       - Check all edges reference valid node IDs
       - Return Vec<String> of errors (empty if valid)

    CRITICAL: Always clone graph before modifying - SqliteGraph doesn't support in-place structural edits. Rewriting creates a new graph.

# Milestone v1.4.0 Requirements: Pub/Sub Enhancements

**Last Updated:** 2026-02-03
**Status:** Planning

## Functional Requirements

### FR-1: KV Prefix Scanning

**ID:** FR-1
**Priority:** High
**Source:** PUBSUB_COMPARISON.md Priority 1.1

The system SHALL provide a method to scan all KV entries with a given prefix:

```rust
// On GraphBackend trait
fn kv_prefix_scan(
    &self,
    snapshot_id: SnapshotId,
    prefix: &[u8]
) -> Result<Vec<(Vec<u8>, KvValue)>, SqliteGraphError>
```

**Acceptance Criteria:**
- [ ] Returns all keys that start with the given prefix
- [ ] Respects snapshot isolation (only entries visible at snapshot)
- [ ] Filters expired entries (TTL check)
- [ ] Works with both SQLite and Native V2 backends
- [ ] Returns results in lexicographic order by key
- [ ] Empty prefix returns all keys
- [ ] CLI command: `sqlitegraph kv scan --prefix <bytes>`

### FR-2: Query Nodes by Kind

**ID:** FR-2
**Priority:** High
**Source:** PUBSUB_COMPARISON.md Priority 1.2

The system SHALL provide a method to query all nodes with a given kind:

```rust
// On GraphBackend trait
fn query_nodes_by_kind(
    &self,
    snapshot_id: SnapshotId,
    kind: &str
) -> Result<Vec<i64>, SqliteGraphError>
```

**Acceptance Criteria:**
- [ ] Returns all node_ids where node.kind == kind
- [ ] Respects snapshot isolation
- [ ] Case-sensitive matching on kind
- [ ] Works with both SQLite and Native V2 backends
- [ ] CLI command: `sqlitegraph nodes by-kind <kind>`

### FR-3: Query Nodes by Name Pattern

**ID:** FR-3
**Priority:** High
**Source:** PUBSUB_COMPARISON.md Priority 1.2

The system SHALL provide a method to query nodes by name pattern:

```rust
// On GraphBackend trait
fn query_nodes_by_name_pattern(
    &self,
```

- [ ] Pattern matching on node.name field
- [ ] Respects snapshot isolation
- [ ] Works with both SQLite and Native V2 backends
- [ ] CLI command: `sqlitegraph nodes by-name <pattern>`

### FR-4: Pattern Subscription Filters

**ID:** FR-4
**Priority:** High
**Source:** PUBSUB_COMPARISON.md Priority 2.2

The `SubscriptionFilter` SHALL support pattern-based filtering:

```rust
pub struct SubscriptionFilter {
    // Existing fields...
    pub kind_patterns: Option<Vec<String>>,  // e.g., ["agent:*", "user:*"]
    pub name_patterns: Option<Vec<String>>,  // e.g., ["msg_index:*"]
}
```

**Acceptance Criteria:**
- [ ] New fields added to `SubscriptionFilter`
- [ ] Pattern matching uses glob syntax
- [ ] Patterns are matched against node.kind and node.name
- [ ] Constructor method: `SubscriptionFilter::kind_patterns(Vec<String>)`
- [ ] Constructor method: `SubscriptionFilter::name_patterns(Vec<String>)`
- [ ] `matches()` method updated to check patterns
- [ ] Existing tests continue to pass
- [ ] New tests for pattern matching

## Non-Functional Requirements

### NFR-1: Performance

- [ ] KV prefix scan: O(N) where N = keys with prefix
- [ ] Query by kind: O(V) where V = total vertices (acceptable for MVP)
- [ ] Pattern matching: O(V * P) where P = pattern count
- [ ] No degradation in existing pub/sub performance

### NFR-2: Backward Compatibility

- [ ] All existing APIs remain unchanged
- [ ] New fields in `SubscriptionFilter` are `Option<Vec<String>>` (default: None)
- [ ] Existing subscriptions continue to work
- [ ] No breaking changes to `GraphBackend` trait (additions only)

### NFR-3: Code Quality

- [ ] 0 compiler warnings
- [ ] All new code has tests (80%+ coverage)
- [ ] Documentation for all new public APIs
- [ ] Examples in MANUAL.md

### NFR-4: Out of Scope

**Explicitly EXCLUDED from this milestone:**
- IPC / Networking (per user: "not IPC/NETWORKING sqlitegraph was not created for networks")
- Event persistence / replay (Priority 2.1 - defer to future milestone)
- Consumer groups (Priority 3.1 - use external MQ)
- Cross-process delivery (Priority 3.2 - use Redis/NATS)

## Cross-Functional Requirements

### XFR-1: Dual Backend Support

All new features MUST work with both backends:
- SQLite backend: Use SQL queries (LIKE for prefix, = for kind, GLOB for pattern)
- Native V2 backend: Use in-memory iteration (HashMap scan)

### XFR-2: MVCC Isolation

All query operations MUST respect snapshot isolation:
- Only return data visible at the given snapshot_id
- Use same visibility rules as existing operations

### XFR-3: CLI Integration

All new features MUST be accessible via CLI:
- Consistent command structure
- JSON output option
- Human-readable help text
